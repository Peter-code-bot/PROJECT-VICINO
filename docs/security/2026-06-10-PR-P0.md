## Summary

Closes audit finding #1 (CWE-269 / CWE-862, CVSS 9.8). A stranger holding only the public
anon key could become an administrator. There were TWO independent doors to admin:

1. **`make_admin(p_email)`** -- a `SECURITY DEFINER` RPC with NO authorization guard and the
   Supabase default EXECUTE grant to anon/authenticated. A direct
   `POST /rest/v1/rpc/make_admin` self-promoted the caller to admin.
2. **Direct writes to `public.user_roles`** -- the role pivot carried the default table
   write grants for `authenticated`, so an attacker could `INSERT` their own admin row via
   PostgREST without touching `make_admin` at all.

Both are closed. Role management is now centralized behind one admin-guarded RPC, and
`user_roles` is read-only for all clients.

## Changes

### Database (applied in Supabase Studio via Camino 2; mirrored as migrations for git)

- **CH-1 -- make_admin guard.** `CREATE OR REPLACE` with an explicit in-body check that the
  caller already holds `admin` (EXISTS against `user_roles`), `REVOKE ALL FROM PUBLIC`,
  `REVOKE EXECUTE FROM anon`, `GRANT EXECUTE TO authenticated`, `SET search_path`.
- **CH-1b -- user_roles lockdown.** `REVOKE INSERT/UPDATE/DELETE/TRUNCATE/REFERENCES/TRIGGER
  FROM anon, authenticated`; `REVOKE SELECT FROM anon`; `FORCE ROW LEVEL SECURITY`; recreate
  `Admin can manage roles` with USING + WITH CHECK `has_role((select auth.uid()),'admin')`.
  Net authoritative grants: `authenticated | SELECT` only.
- **CH-1c -- manage_user_role RPC.** Admin-guarded `SECURITY DEFINER` function
  `manage_user_role(p_user_id, p_role, p_action)` that is the ONLY write path for roles
  (assign | remove). Protects the last admin on remove with a `FOR UPDATE` lock on the admin
  rows (TOCTOU-safe). `REVOKE` anon/PUBLIC, `GRANT` authenticated, `SET search_path`.

Migrations:
- `supabase/migrations/20260610000001_hotfix_make_admin_and_user_roles_privesc.sql` (CH-1/1b)
- `supabase/migrations/20260610000002_manage_user_role.sql` (CH-1c)

## Smoke evidence

CONFIRMED in Studio:
- **Attacker blocked (direct write).** `INSERT INTO user_roles` as an authenticated non-admin
  -> `permission denied for table user_roles` (grant-level, before RLS).
- **make_admin / manage_user_role attacker.** Non-admin call -> `forbidden` (42501).
- **Admin operates.** An admin assigns/removes a role via `manage_user_role` -> success.
- **Last-admin guard (race-safe).** Removing the only admin raises `no se puede quitar el
  ultimo admin`. The lock uses a plain `PERFORM 1 ... FOR UPDATE` + a separate `SELECT
  count(*)` -- the literal `count(*) ... FOR UPDATE` is invalid in Postgres (SQLSTATE 0A000)
  and is NOT used.

PENDING (must be green before merge -- see below):
- **No-recursion under FORCE RLS.** As an authenticated session, a `has_role`-gated read
  (e.g. `SELECT id FROM products_services LIMIT 1`) must NOT raise
  `infinite recursion detected in policy for relation user_roles`.

---

## Risk and rollback

- **FORCE RLS depends on `postgres` having BYPASSRLS** (Supabase default). If the recursion
  smoke fails, run `ALTER TABLE public.user_roles NO FORCE ROW LEVEL SECURITY;` -- the
  grant-level REVOKEs alone still fully block the privesc; FORCE is defense-in-depth.
- **TOCTOU on last-admin** is fixed live (v2): a plain `PERFORM 1 ... FOR UPDATE` lock on the
  admin rows + a separate `SELECT count(*)`. The literal `count(*) ... FOR UPDATE` is invalid
  in Postgres (SQLSTATE 0A000) and is NOT used. git mirror aligned to the live body.
- **All migrations are idempotent** (CREATE OR REPLACE / DROP POLICY IF EXISTS / REVOKE-GRANT
  / FORCE RLS). RPCs do `ON CONFLICT DO NOTHING` (assign) so re-runs are safe.
- **No data migration**; rollback affects access-control evaluation only, not stored rows.
- Bootstrap: the first admin is seeded once by a direct `INSERT INTO public.user_roles
  (user_id, role) VALUES ('<admin_uuid>', 'admin')` run as `postgres` (make_admin's guard
  rejects a NULL auth.uid(), so it cannot bootstrap the first admin).
