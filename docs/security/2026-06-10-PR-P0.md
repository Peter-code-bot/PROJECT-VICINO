## Summary by finding

- **#1 (CVSS 9.8, P0) privilege escalation** -- `make_admin` was SECURITY DEFINER with no guard
  and anon EXECUTE; `user_roles` had default authenticated writes. Both let a stranger become
  admin. Fixed: admin-guarded `make_admin`, `user_roles` write-lockdown + FORCE RLS, and a single
  `manage_user_role` RPC (the only role-write path; race-safe last-admin guard).
- **#4 (8.1) chat RPC BOLA/IDOR** -- `get_or_create_chat` / `mark_messages_as_read` trusted a
  client-supplied identity. Fixed: derive the actor from `auth.uid()` (signature-preserving, zero
  app edits) + participation guard + REVOKE anon.
- **#5/#6/#7 (9.1/8.1/7.5) mass-assignment** -- anon/authenticated had table-wide UPDATE on
  profiles/products/reviews/sale_confirmations, allowing column mass-assignment (is_verified,
  trust_*, stats, is_hidden, confirm flags...). Fixed: column-level GRANT allowlists; per-actor
  sale RPCs (`confirm_sale`/`cancel_sale`); stat triggers -> SECURITY DEFINER; admin-moderation
  RPCs (`moderate_set_content_hidden`/`moderate_review`); a `increment_product_view` RPC for the
  view counter.
- **#2 (8.6) PII column exposure** -- anon/authenticated could read profiles PII columns. Fixed:
  REVOKE SELECT on `email, telefono, rfc, ubicacion_lat, ubicacion_lng, fcm_token`; self PII via
  `get_my_profile()`, admin PII via `admin_list_users()`/`admin_get_user()`. user_id (public
  handle) and is_hidden (moderation) stay readable.
- **Credential incident** -- the push-on-sale DB trigger had a service_role JWT hardcoded in
  plaintext. Eradicated: rotated the key, moved it to Vault, replaced the trigger with
  `call_send_push_on_sale()` (reads Vault); redacted the token committed in a prior audit doc.

## Smoke evidence (all CONFIRMED in Studio)

- **#1 attacker** -- authenticated direct `INSERT user_roles` / `make_admin` -> `permission denied`
  / `forbidden`. Admin role mgmt works via `manage_user_role`. Last-admin removal blocked
  (FOR UPDATE, no TOCTOU).
- **FORCE RLS recursion (P0 gate)** -- a has_role-gated read does NOT recurse (`via_rls=false`,
  `postgres.rolbypassrls=true`). Confirmed safe.
- **#4** -- chat RPCs are `authenticated|EXECUTE` only; non-participant mark-read -> forbidden.
- **#5/#6/#7** -- direct PATCH of is_verified / vistas_count / sale flags / review visible ->
  `42501`; legit writes (foto/fcm_token, product allowlist, respuesta, confirm own flag) succeed;
  sale completion + rating recompute still fire (stat triggers DEFINER).
- **#7 moderation** -- moderation hide/approve/resolve work via RPCs; the guard is admin-only for
  profile/user (suspend) and review (hide/approve), admin|moderator for listing/message.
- **#2** -- anon/auth `select=email,...` -> `42501`; self gets own email via `get_my_profile`;
  admin panels show emails; public seller page (user_id) + search + moderation (is_hidden) work.
- **Incident** -- `call_send_push_on_sale` reads Vault (no literal JWT); `send-push` returns 400
  (valid rotated token), not 401. `rg eyJhbG` clean of real tokens.

## Risk & rollback
- All migrations idempotent (CREATE OR REPLACE / DROP POLICY IF EXISTS / REVOKE-GRANT / FORCE RLS /
  ALTER FUNCTION). RPCs `ON CONFLICT DO NOTHING` / participant-guarded.
- FORCE RLS on user_roles: safe because `postgres` has BYPASSRLS (confirmed). Rollback:
  `ALTER TABLE user_roles NO FORCE ROW LEVEL SECURITY;` (grant REVOKEs still block the privesc).
- No data migration; rollbacks affect access-control evaluation only.
- Reconciliation notes (RESOLVED): user_id/is_hidden re-granted (#2); moderation guard narrowed to
  app authz (#7e); both mirrored + self-healing.
